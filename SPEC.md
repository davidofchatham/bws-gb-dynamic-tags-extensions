# SPEC — `src:site` unified site source (v1.9.0, Stage A)

Source: `.claude/plans/src-site-unified-source.md`. Scope: **Stage A only** (base-tag `src:site`). Stage B (#26 try_ slot derive) + Stage C (slot dispatch arm, `class-site.php`) OUT — not in this spec.

## §G — Goal

One source `src:site` for all site-wide data behind existing base tags (text/permalink/image/content/datetime_single/datetime_range). One tag name, `use:` enum picks datum. Unlocks `{{option}}`-style key reads on datetime (ACF date opts) + content (block-markup opts) — combos GB Pro lacks.

## §C — Constraints

- C1 — No new source class, no registry registration in Stage A (decision S2). Registering pulls site into `get_effective_sources()` (try_ slot map) before Stage C dispatch arm exists → `resolve_id()`→`post_fn(0)` footgun. Class deferred to B/C.
- C2 — `site` works as `bws_base_source_option()` dropdown value + early-gate string alone. `bws_resolve_post_by_source` never reached for site (early gate short-circuits before it).
- C3 — Option reads OUR responsibility to allowlist-gate. `Meta_Handler::get_option()` enforces only blocklist (`DISALLOWED_KEYS`); does NOT consult `generateblocks_dynamic_tags_allowed_options` (that lives in GB Pro `get_option` callback `class-register.php:257-296`, upstream, skipped when calling handler direct).
- C4 — Allowlist seed = **GB Pro parity** (REVISED, was empty A2 — see B2). GB Pro's `get_option` callback (verified, GB Pro 2.6.0-beta.2 `class-register.php:268-291`) seeds: `siteurl, blogname, blogdescription, home, time_format, user_count` PLUS `array_keys( GenerateBlocks_Pro_Dynamic_Tags_ACF::get_instance()->get_acf_option_fields() )` (every registered ACF options-page field — registration IS the opt-in), then `apply_filters('generateblocks_dynamic_tags_allowed_options', merged)`. We mirror this EXACTLY. ACF-options fields auto-allow (no manual filter). Empty seed (old A2) was a misread of GB Pro that blocked ACF option fields → site option reads returned empty (B2). `{{...src:site|key:blogname}}` now renders out-of-box like `{{option key:blogname}}`.
- C5 — Datetime-site = ACF-options read (`get_field($key,'option')`), NOT `get_option`. ACF retains `return_format`; `get_option` loses it. Acceptable — datetime tags already ACF-bound everywhere, no NEW lock-in. Non-datetime site values stay ACF-free.
- C6 — No `meta`/`single_only` control on site `key`. `get_option` called `$single_only=true` (matches Pro). Array-root reads (no subkey) → empty; useful reads = scalar leaf via dot-path.
- C7 — `linkTo` static value list this release (L3a). Conditional JS gates whole controls by `element.key`, not individual `options[]` entries → dropdown shows all 4 values every context. Leakage harmless both ways. Per-value `show_if` gating = L3b, deferred (general capability → #27), do NOT block v1.9.0.
- C8 — Content pipeline UNCHANGED. `bws_render_block_content` shipped 1.8.0; Stage A = call site only, no refactor.
- C9 — Text domain `'generateblocks'`; all fns prefixed `bws_`; version 1.9.0.

## §I — Interfaces

- I.src — `bws_base_source_option()` (base-tags.php ~557): add `{value:'site', label:'Site'}`.
- I.resolve — new `bws_site_resolve_value($tag, $options, $instance)`. Used by text/title/permalink/image/content ONLY (not datetime). **NO `use:option` value** (dropped, B3). Resolve order: (1) named site `use:` value → its datum; (2) else → site option key-read. This is NOT "use unset = key" globally — it is `src:site` specifically routing any non-named-site `use` (including each tag's own default, e.g. content's `content`) into the option read, because under `src:site` there is no post entity to apply a post-default to.
  - NO site `use` enum on any tag (V9 — dropped `logo`/`home_url`/`site_url` as redundant with bare analogs; `site_url()` intentionally unreachable this release). Site model = **bare → best analog, `key` → option**.
  - Per-tag bare analog (no key): title→`get_bloginfo('name')`; text→`get_bloginfo('name')`? NO — text keyed-by-nature, bare=empty (V9); content→`get_bloginfo('description')` (B4); permalink→`home_url()`; image→logo: `(int)get_theme_mod('custom_logo')` → `bws_get_attachment_data($id, as ?? 'url', size ?? 'full')` (empty/no logo → '').
  - title tag has NO `use` enum → `$tag==='title'` (or `use:title`) → `get_bloginfo('name')`.
  - content site PARALLEL (B4): bare `{{content src:site}}` (no `key`) → `get_bloginfo('description')` (tagline = blogdescription), paralleling post→content / term→description / site→description. content with `key` set → option key-read (rich-option render path). So content's named site datum IS the description; `tagline` is therefore REMOVED from the text `use` enum (single-path tagline killed — reach it via `{{content src:site}}`).
  - option key-read (the non-named branch) + `key` set → allowlist-gate (C4 GB-parity), then `Meta_Handler::get_option($key,true,'')`. Content: pass raw through `bws_render_block_content($raw,'option:'.$key)` instead (NO extra `wp_kses_post`). Empty `key` → for content = tagline (above); for text/permalink/image = `''`.
- I.gate — text/content/title/permalink/image callbacks (base-tags.php 725/797/841/898/932): early gate `if ('site'===($options['src']??'')) return bws_site_resolve_value(...)` before entity-resolve. title callback site path must still apply `bws_wrap_with_link(...,1,'site')` (link-eligible).
- I.dt — `bws_base_datetime_single_callback` (datetime-tags.php:988) / `_range_callback`: SEPARATE path. On `src===site` call `bws_datetime_single_core('option', bws_base_map_datetime_options($options), $instance)` (+range equiv), then `bws_wrap_with_link(...,1,'site')` tail. NO `use` enum — `key` is direct field-key control.
- I.read — **DT-1**: `bws_read_field` (field-helpers.php:~230, before `return null` tail) add: `if('option'===$post_id && function_exists('get_field')){ allowlist-gate $key; return get_field($key,'option'); }`. Sentinel currently dead-ends at null → zero behavior change for int/loop/term callers.
  - **DT-1b (caller seam, →B1/V7)**: `bws_parse_combined_date_time` (datetime-helpers.php:272/280) passes `$numeric_id` to `bws_read_field`, NOT raw `$post_id`. `$numeric_id = is_numeric($post_id)?(int):false`, so `'option'`→`false` and the DT-1 sentinel never arrives. MUST thread the `'option'` sentinel to the value read: `$value_id = ('option'===$post_id) ? 'option' : $numeric_id;` then pass `$value_id` at both date+time reads. Non-`'option'` callers unchanged (still `$numeric_id`).
- I.link — **L3**: `bws_resolve_link_url` (link-helpers.php:37) add `'site'` entity branch: `linkTo:site`→`home_url()` (ignores `$id`); `linkTo:key`+site→allowlist-gate `$link_key` then `get_option($link_key)`. Site callbacks pass `$link_type='site'`, sentinel `$link_id=1` (defeats `if(!$id)` guard line 38). Add `site` value to `linkTo` dropdown (`bws_get_link_options`). Link options on text/title/datetime_* only (`supports_link_wrap`); content/permalink/image excluded.
- I.opts — per-tag `use:` enum. **NO `option` value anywhere** (B3). Site named-data are `use:<not key>` entries; the default key-mode (empty `use`) + `key` set = option read.
  - NO site `use` values on any tag (V9). text: `use` enum = existing `title`,`key` (no site-specific value; bare text+site = empty, keyed) | permalink: NO `use` enum at all (was site-only `site_url`/`home_url`, both dropped — bare = `home_url()`) | image: `use` enum = existing `featured`,`key` (NO `logo`; bare image+site = logo) | content: NO site `use` value — bare = description (B4), `key` = option | title: NO use enum (site→name) | datetime_single/range: NO use enum.
  - `key` shown for site whenever key-mode: extend each tag's existing `key` `show_if` to also show under `src:site` in key-mode. text key `show_if not_in:title,tagline` already covers it (empty/key/option-namespace all key-mode). permalink key `show_if {src:site, use:not_in:site_url,home_url}`. image key `show_if not_in:featured,logo`. content key `show_if in:key` PLUS site (use empty under site = content default is post-content; site needs key → see note). title + datetime: datetime existing `key` always-visible (direct field/option key); title no key.
  - NOTE content: non-site content default `use` = `content` (post content); under `src:site` there is no post content, so site-content MUST be key-mode. Resolver treats `src:site` content as option read regardless of `use` (content has no named site datum). `key` `show_if` for content: `show_if_any {use:key, src:site}`.
  - Suppress for site (`show_if src:not:site`) — on ALL site-capable tags incl. title:
    - `srcTermIn` — conceptually N/A for site (no entity to hop terms from).
    - `ref` — suppressed in Stage A because there is NO site→ref wiring yet (early gate short-circuits before `bws_resolve_post_by_source`; resolver switches on `use` only). NOT "ref never applies to site" — ACF relational fields stored as site options are plausible (untested). Re-expose `ref` (drop the gating) when a site→ref resolution path ships → **#28**. (Mechanically `ref` keeps its existing `show_if src:ref` which already hides it for site — no new gate needed.)
- I.adr — [ADR 0001](docs/adr/0001-site-option-read-allowlist.md): **REWRITTEN** (B2) — GB-parity allowlist (6 WP defaults + ACF option-field keys + filter), "OUR resolver gates not handler" (still true), 3 gated read paths (site option key-mode, site `linkTo:key`, datetime `get_field(…,'option')`). Empty-seed premise was a misread of GB Pro — corrected.

## §V — Invariants

- V1 — `bws_site_allowlist_ok($key)` MUST gate every option read with the **GB-parity seed** (REVISED, B2): `$seed = ['siteurl','blogname','blogdescription','home','time_format','user_count']; if (class_exists('GenerateBlocks_Pro_Dynamic_Tags_ACF')) $seed = array_merge($seed, array_keys(GenerateBlocks_Pro_Dynamic_Tags_ACF::get_instance()->get_acf_option_fields())); $allowed = apply_filters('generateblocks_dynamic_tags_allowed_options', $seed); $parent = explode('.',$key)[0]; return in_array($parent,$allowed,true);`. NOT empty seed (that misread of GB Pro blocked ACF option fields, B2). !C3,C4,ADR 0001 — `@invariant` PHPDoc on `bws_site_allowlist_ok`.
- V2 — All THREE option-read paths gate through `bws_site_allowlist_ok` (same GB-parity filter): site option key-mode read, site `linkTo:key`, datetime `get_field(…,'option')`. Consistent disclosure boundary. ACF keys flat → gate whole `$key` (dot-path split only matters for wp_options arrays; `$parent` = first segment handles both).
- V3 — Datetime-site output formatting MUST flow through `bws_build_single_format` / `bws_build_range_format`. Site-format fallback (`get_option('date_format')`/`time_format`) = their tier-3; MUST NOT be re-implemented independently. Format precedence: (1) tag `custom_format` if `format_type:custom` → (2) ACF `combined_format` → (3) site format. !C5,I.dt — reason DT-1 (reuse `_core`) beats DT-2 (dup formatting regresses chain).
- V4 — Stage A introduces NO source class + NO registry registration. `site` is dropdown-value + early-gate-string only. !C1,C2.
- V5 — Site early gate MUST pre-empt any `SourceRegistry::get_source('site')` lookup (returns null → choke). Preview-label + GB-registration paths must not call it for site. Editor preview = likely edge.
- V6 — `content` site option-read recursion bounded by existing 1.8.0 guard via `'option:'.$key` cache key (key collision = guard hit, by design). Distinct keys bounded by `bws_content_max_recursion_depth` (default 3). No pipeline change. !C8.
- V7 — Site-datetime value read MUST receive the `'option'` object-id sentinel. `bws_parse_combined_date_time` passes a value-read id to `bws_read_field` that is `'option'` when `$post_id==='option'`, else `$numeric_id` (existing behavior). The format-lookup path (`$acf_object_id` via `bws_resolve_acf_object_id`) already resolves `'option'` correctly; the VALUE-read path is a separate arg and was NOT threading it (B1). !I.read,DT-1b — drove B1.
- V9 — **Source-analog resolution.** SCOPE: base-tag callbacks and try_ **slot 1** only — i.e. wherever an empty `use`/`key` wire value means "the template default" (the strip-default-first-value position). Does NOT govern try_ slot ≥2: there an empty wire = "same as previous slot" (carry-forward to a concrete inherited token, see V8), which never re-derives an analog. Within scope: when a tag resolves to its default (empty key, default `use` token) it SHOULD yield the best intrinsic analog datum for the active source, WHERE ONE EXISTS. Analogs (current): title→{post title, term name, site name}; content→{post content, term description, site description}; permalink→{post URL, term URL, site `home_url()`}; image→{post featured, —(term has none — gap, #29 may fill via configurable default key), site logo}; text→keyed by nature in ALL contexts (default = empty, key required). Where a source has no intrinsic analog (term image today, site text-body), default = empty + key required (honest gap). Named `use:`/`key` are OVERRIDES/escape-hatches, not the primary path. COROLLARY: a named `use:` value that would duplicate the in-scope default analog is redundant and SHOULD NOT exist (drove dropping site `use:logo`/`use:home_url`/`use:site_url`). NOTE: a concrete analog default (e.g. site image→logo) is also what try_ slot ≥2 carry-forward inherits, so concrete > empty for slot reuse. Image bare-default policy (native-vs-keyed per source) still being finalized — see open discussion; site image→logo is provisional. !I.resolve,I.opts,B4,#29. → migrate to CONTEXT.md post-ship.
- V10 — **`use`/`key` control labels track source scope.** Adding a new source type that routes through an EXISTING control (notably the `use:key` field-key path and the `key` control) widens what that control covers, so its LABEL must be reconsidered/adjusted. E.g. `use:key` entry labeled "Meta/Custom Field" (base-tags.php:67,244,319 + content "Custom Content Field" :124,356) is accurate for post/term meta but undercounts `src:site`, where the SAME `key` reads a wp_options/ACF-options value. PROCESS: every new source type addition MUST include a label-review pass over shared `use`/`key` controls; a stale label that omits a now-valid source is a defect. → migrate to plugin-integration.md "Registering an External Source" checklist post-ship.
  - **V10a — DYNAMIC entry labels (chosen approach, deferred build → T17).** Rather than ONE long static label naming every source ("Meta/Custom/Option Field"), make the `use` dropdown ENTRY label src-aware so it names only the active source's meaning (shorter, contextual). FEASIBILITY VERIFIED: GB free fires `applyFilters('generateblocks.editor.tagSpecificControls', createElement(g,{key,...m}), allOptions, {state,setState})` (generateblocks-2.3.0-beta.2 dist text/index.js) — the select's entries are `element.props.options[]`, mutable via `wp.element.cloneElement(element, { options: rebuilt })`. MECHANISM: add per-value `label_by` map on `options[]` entries (`['value'=>'key','label'=>'Meta/Custom Field','label_by'=>['site'=>'Field / Option']]`); extend `assets/js/editor-conditional-options.js` filter to, after show/hide, rebuild `options` picking `entry.label_by[state.src] ?? entry.label`. Entry labels are the priority target (higher value than the `key` control label). Declarative, lives with the option like `show_if`.
  - **V10b — entry FILTERING shares this seam (= #27 / L3b).** Filtering `use` entries by `src` (show only source-valid values) uses the SAME `cloneElement(element, { options: rebuilt })` rewrite — relabel rewrites each entry's `label`, filter drops invalid entries; one loop over `element.props.options[]` does both. Feasibility verified the same way (control reads `options` from props; our returned element is terminal — GB does not re-derive). So building T17 (relabel) makes #27 (filter) near-free. TOGETHER they yield a per-source CONTEXTUAL `use` enum: one entry shown+labeled per active source (e.g. a single "native image" entry → "Featured"/"Site Logo"/"Avatar" by src, hidden where no native exists). The PARKED image-model decision (native-vs-keyed, `use:featured` slot) should be made ASSUMING this capability exists — it's verified-cheap, decoupled from the JS build timing. Build with T17/#27 post-core.
- V8 — NO `use:option` value may exist on any base tag (option IS a key-read, namespaced by `src` not `use` — adding it broke the convention, B3). Each base tag's `use` default is its OWN (text/image → `key`; content → `content`/post-content; permalink/title → none) — "`use` unset" does NOT universally mean key-mode. Site option-read is resolved PER TAG inside `bws_site_resolve_value`: named site `use:` values map to named data; everything else (incl. each tag's own non-key default, e.g. content's `content`) falls to the option key-read **only because `src:site` forced that branch** — the resolver special-cases content (`$use=''`) since content has no named site datum. NOT a claim that "`use` unset = key" globally. **try_ slots:** in try_ slot ≥2 an empty `use`/`src` means "same as previous slot" (carry-forward), a THIRD meaning — V8 governs base tags only; try_ semantics are Stage B/C (#26). !B3,I.resolve,I.opts.
