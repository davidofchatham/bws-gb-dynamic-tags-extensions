# SPEC — `src:site` unified site source (v1.9.0, Stage A)

Source: `.claude/plans/src-site-unified-source.md`. Scope: **Stage A only** (base-tag `src:site`). Stage B (#26 try_ slot derive) + Stage C (slot dispatch arm, `class-site.php`) OUT — not in this spec.

## §G — Goal

One source `src:site` for all site-wide data behind existing base tags (text/permalink/image/content/datetime_single/datetime_range). One tag name, `use:` enum picks datum. Unlocks `{{option}}`-style key reads on datetime (ACF date opts) + content (block-markup opts) — combos GB Pro lacks.

## §C — Constraints

- C1 — No new source class, no registry registration in Stage A (decision S2). Registering pulls site into `get_effective_sources()` (try_ slot map) before Stage C dispatch arm exists → `resolve_id()`→`post_fn(0)` footgun. Class deferred to B/C.
- C2 — `site` works as `bws_base_source_option()` dropdown value + early-gate string alone. `bws_resolve_post_by_source` never reached for site (early gate short-circuits before it).
- C3 — Option reads OUR responsibility to allowlist-gate. `Meta_Handler::get_option()` enforces only blocklist (`DISALLOWED_KEYS`); does NOT consult `generateblocks_dynamic_tags_allowed_options` (that lives in GB Pro `get_option` callback `class-register.php:257-296`, upstream, skipped when calling handler direct).
- C4 — Allowlist seed = **GB Pro parity** (REVISED, was empty A2 — see B2). GB Pro's `get_option` callback (verified, GB Pro 2.6.0-beta.2 `class-register.php:268-291`) seeds: `siteurl, blogname, blogdescription, home, time_format, user_count` PLUS `array_keys( GenerateBlocks_Pro_Dynamic_Tags_ACF::get_instance()->get_acf_option_fields() )` (every registered ACF options-page field — registration IS the opt-in), then `apply_filters('generateblocks_dynamic_tags_allowed_options', merged)`. We mirror this EXACTLY. ACF-options fields auto-allow (no manual filter). Empty seed (old A2) was a misread of GB Pro that blocked ACF option fields → site option reads returned empty (B2). `{{...src:site|key:blogname}}` now renders out-of-box like `{{option key:blogname}}`.
- C5 — Datetime-site = ACF-options read (`get_field($key,'option')`), NOT `get_option`. ACF retains `return_format`; `get_option` loses it. Acceptable — datetime tags already ACF-bound everywhere, no NEW lock-in. Non-datetime site values stay ACF-free.
- C6 — No `meta`/`single_only` control on site `key`. `get_option` called `$single_only=true` (matches Pro). Array-root reads (no subkey) → empty; useful reads = scalar leaf via dot-path.
- C7 — `linkTo` static value list this release (L3a). Conditional JS gates whole controls by `element.key`, not individual `options[]` entries → dropdown shows all 3 values (none/permalink/key) every context. Leakage harmless both ways. Per-value `show_if` gating = L3b, deferred (general capability → #27), do NOT block v1.9.0. NO `linkTo:site` value — site permalink-analog folds into `linkTo:permalink` (V-link).
- C8 — Content pipeline UNCHANGED. `bws_render_block_content` shipped 1.8.0; Stage A = call site only, no refactor.
- C9 — Text domain `'generateblocks'`; all fns prefixed `bws_`; version 1.9.0.
- C10 — **Qualifying gate for new `use:` values (B7) — TWO-SIDED.** A new named `use:` value (or per-source analog) MUST satisfy AT LEAST ONE of two independent tests; reject ONLY if it fails BOTH. (1) **Uniqueness** — offers an affordance no existing path gives: a datum unreachable elsewhere (`src:ref` excerpt: no native GB path) OR a transform/traversal that adds value (datetime format chain). (2) **Strong cross-source analog** — fills the SAME conceptual slot as the tag's analogs in other sources, so the bare tag "just works" per context (title→{post title, term name, site name}; content→{post content, term description}); this qualifies EVEN IF the datum is ALSO reachable via `key`/GB-native, because the value is the consistent mental model, not a path the user lacked (it IS the design principle, restated as a gate). A value that fails BOTH (datum already reachable AND no transform AND no cross-source slot) is proliferation → reject — that is tagline (GB `{{site_tagline}}` covers it, nothing to format, AND no slot it parallels — a one-off site datum, not an analog). Process gate (decision-time, per candidate), NOT a runtime invariant. Cost acute until cross-token filtering (V10b/#27) ships: no per-source entry filter → every value hangs on the base tag in EVERY source. "Feeds a multi-slot tag" (base `text` enum → `try_text`) is NOT sufficient — overloads the base tag as a feeder; fix is decoupling the multi-slot feed (#26 + cross-token), so a value clearing the gate ONLY as multi-slot feedstock is DEFERRED, re-evaluated when the decouple ships. Worked examples + verdicts live in `docs/tag-reference.md` §Qualifying test. !V9,B7,#26,#27.

### §V — `{{email}}` tag (added; full design in `.claude/plans/email-tag.md`)

- VE1 — **`noLink` is an inverted bare-key boolean; mailto wrap is DEFAULT-ON.** `{{email}}` wraps its address in `mailto:` UNLESS the `noLink` bare key is present. Absence = wrap (default-on); `noLink` present = plain text. MUST NOT be modeled as a positive `linkWrap`/`link:true` default: GB's generic custom-option serializer DROPS `false` (`DynamicTagSelect.jsx:559-562` — `false===value`→no token) and only emits bare keys for `true` (`:564-566`); the hardcoded `required:false` branch (`:549-551`) is bespoke and NOT hookable. So "default-on, serialize-when-off" is reachable ONLY by an inverted-name bare key. `noLink` is the final design (not a stopgap) — there is NO deferred positive `linkWrap:false`. Render: link-on iff `noLink` absent from options. `@invariant` PHPDoc on the email callback. → migrate to PHPDoc + CONTEXT.md post-ship.
